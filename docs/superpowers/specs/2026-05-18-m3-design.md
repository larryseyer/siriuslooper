# M3 — Channel-driven tape allocation + ProcessingChain + TapeWriter — Design

**Date:** 2026-05-18
**Scope:** The first behaviour-bearing milestone in Part B of the V7 alignment plan. Replaces the M2 assert-false stubs in `InputMixer` with real bodies, lands the first `Tape<T>` instances in product code, introduces a dedicated tape-writer thread fed by a lock-free SPSC queue, makes channel-driven tape topology real, and adds per-`SignalType` processing chains (no-op bodies; real DSP arrives in M5/M9/M12/M13). Implements V3 Steps 3–5 of the transition guide.
**Status:** Brainstorm complete; awaiting implementation plan.
**Plan reference:** `docs/superpowers/plans/2026-05-17-v7-alignment.md` lines 263–321 (M3).

## Problem

M2 shipped the V3 mixer vocabulary — `SignalType`, `TapeMode`, `Channel`, strong-typed `InputId` / `ChannelId`, and `InputMixer` / `OutputMixer` skeleton classes whose bodies all `assert(false && "M3-M5 stub")`. The type shapes are in place, but nothing yet *does* anything. The product code still has zero `Tape<T>` instances; tapes exist only in test code.

M3 turns the architecture from "compiles, doesn't run" into "compiles, runs, writes tapes." The audio thread must walk channels, apply processing, and route processed samples to disk *without* allocating, blocking, or doing synchronous I/O on the audio thread. A dedicated writer thread does the actual file work. Channel-driven tape allocation finally exists: one input no longer maps to one tape — channels declare their own tape mode (`CommitToTape` / `NonDestructive` / `NoTape`) and the mixer allocates accordingly.

This milestone is the foundation M4 (Direct Layer), M5 (Output Mixer real bodies), M9 (MIDI), M12 (Video), and M13 (File I/O) all build on. Getting the audio-thread / writer-thread split right here pays off every time another modality wires into the same plumbing.

## Design principle: real-time safety is non-negotiable

`docs/RT_SAFETY_CONTRACT.md` (in repo) is the audit checklist every audio-thread change must pass. M3's central design constraint: the audio thread never allocates, never blocks, never does I/O. The dedicated writer thread is the only thing that touches the filesystem. The SPSC queue between them is fixed-capacity; back-pressure is *visible* via `OverloadProtection`, never silent.

A failed-to-enqueue buffer is dropped and reported, not retried or stashed. A writer-thread I/O failure is logged and surfaced, not propagated up to the audio thread. The two threads are decoupled at every boundary the contract names.

## Operator decisions locked in this brainstorm (2026-05-18)

Five decisions were locked through the brainstorm. They are listed here so the implementation plan reads them as source-of-truth, not as recommendations.

1. **`ProcessingChain` shape — abstract base + four concrete subclasses.** `ProcessingChain` is a virtual interface; `AudioChain`, `MidiChain`, `VideoChain`, `FileChain` inherit. `Channel` holds `std::unique_ptr<ProcessingChain> processing`. Virtual call cost is one indirection per buffer — negligible. Matches V3 transition guide §2.1's per-`SignalType` `ChannelStrip` enumeration.
2. **SPSC queue payload — inline fixed-size POD per message.** `TapeWriteMessage { ChannelId id; Rational lmcTime; std::size_t sampleCount; std::array<std::byte, kMaxTapeWriteMessageBytes> samples; }`. Self-contained; the audio thread memcpys processed samples into the message and enqueues. No pointer-into-ring complications; ownership is trivial. `kMaxTapeWriteMessageBytes = 32 * 1024` (32 KB → 4096 stereo float32 samples — headroom for any reasonable buffer × 2 channels at the upper end of `EngineConfig::preferredBufferSize`).
3. **Per-input flag ownership — descriptor carries initial values, mixer holds runtime state.** `InputDescriptor` gains `bool rawDirectMonitor`, `bool enabled`, `ChannelDefaults defaults` as initial-value fields. `InputMixer::registerInput` copies them into the mixer's runtime state. Subsequent setters mutate the mixer only; descriptor stays immutable value-typed metadata (ready for SAF persistence in M11).
4. **SPSC overflow policy — drop + `OverloadProtection.reportLoad(1.0)`.** When the writer-thread queue is full, the audio thread drops the message and reports overload through the existing M1 Session 3 surface. Never blocks. Diagnostics pane's shed-count already exposes this to the operator.
5. **NonDestructive parameter-tape format — JSONL placeholder superseded by M11.** M3 ships a minimal one-event-per-line shape (`{"t": <lmcRational>, "param": "<name>", "value": <number>}`). M11 owns the canonical JSONL schema for all parameter/control/system tapes (plan line 755). M11 must either read M3-era partials losslessly or ship a one-time migration shim — see plan amendment below.

## Architecture

### Audio-thread path (`InputMixer::processBuffer`)

1. Walk active channels in registration order.
2. For each channel, invoke `channel.processing->process(...)` — a virtual call on the abstract `ProcessingChain` base. Audio chains are a no-op in M3 (real DSP in M5); MIDI/Video/File chains pass through.
3. For channels in `CommitToTape` or `NonDestructive` mode:
   - Construct a `TapeWriteMessage` on the audio thread's stack.
   - `memcpy` the processed samples into the inline `samples` array.
   - Stamp `lmcTime` from the current callback's LMC time.
   - Try to enqueue on the shared `LockFreeSpscQueue<TapeWriteMessage>`.
   - On queue-full: call `overloadProtection_.reportLoad(1.0f)` and return. No retry.
4. Channels in `NoTape` mode are walked but never enqueue — they exist for Direct Layer routing only.

The whole loop is bounded, allocation-free, lock-free. Verified by Session 2's RT-safety instrumentation tests.

### Writer-thread path (`TapeWriter`)

1. Owns one `std::thread`; sleeps on the SPSC queue's condition variable.
2. On wakeup, dequeues messages and routes by `ChannelId`:
   - **`CommitToTape`** — append `samples[0..sampleCount]` to `<sessionUuid>/<tapeId>.tape.partial`.
   - **`NonDestructive`** — append dry samples to one `.tape.partial` (matching M3's understanding of "dry" — see §"What 'dry' means in M3" below) and append a JSONL parameter-delta event to the parallel `<tapeId>.params.partial`.
3. Flushes at the tier-appropriate interval pinned in plan line 319: Lavish per-buffer, Comfortable 50 ms, Tight 200 ms, Survival 1000 ms. Drives the interval from `CapabilityTier`.
4. On dtor: sets atomic `shouldExit_`, notifies the cv, joins. The thread drains the queue before exiting so no in-flight samples are lost.
5. **I/O error handling** — disk full / permission denied / file-not-writable: catch, increment a per-channel error counter, log via `juce::Logger`, surface via `OverloadProtection.reportLoad(1.0)` (semantically "engine can't keep up"). The channel keeps trying on subsequent buffers — recoverable, not fatal.

### Tape finalization (`.tape.partial` → `<sha256>.tape`)

The finalize trigger is `InputMixer::finalizeChannel(ChannelId)`. It is the M3 API surface for "this channel's recording is complete; promote its partial files to immutable content-addressed tapes." The call:
1. Signals the writer thread to flush pending writes for the channel.
2. Closes the `.tape.partial` file(s).
3. Computes sha256 of the closed file(s).
4. Renames `<sessionUuid>/<tapeId>.tape.partial` → `<contentRoot>/<sha256>.tape` (and the params sibling if present).
5. Registers the resulting `TapeId` in `TapeStore`.

This preserves the content-addressed immutability invariant the existing `TapeStore` guarantees: the only thing that ever lands at the content-addressed path is a fully-formed, hashed, immutable blob. Partial files live in a separate working directory and never collide with the content store.

The *user-facing trigger* (arm/disarm gesture, automatic on session save, etc.) is wired in M22 — see plan amendment §1 below.

### What "dry" means in M3

NonDestructive mode is specified as "tape contents = dry channel input + parameter tape carrying the processing delta" (plan line 276). With Audio chains being no-op in M3, "processed" == "dry" today, so the parameter-delta tape carries no meaningful deltas. M3 still writes the JSONL event stream (an empty event sequence is a valid sequence); the format earns its first real entries in M5 when Audio chains gain real DSP. This is acceptable: the storage shape is correct, the round-trip works, the data is just trivially empty until M5.

## Components

### `engine/include/ida/ProcessingChain.h` + `.cpp` (NEW, Session 1)

```cpp
namespace sirius {

class ProcessingChain {
public:
    virtual ~ProcessingChain() = default;
    virtual SignalType signalType() const noexcept = 0;
    // process() signature is SignalType-specific; each subclass declares its own
    // typed entry point. The base class exists as the polymorphic anchor on
    // Channel; callers down-cast via signalType() before calling process().
};

class AudioChain  : public ProcessingChain { /* SignalType::Audio; no-op process() */ };
class MidiChain   : public ProcessingChain { /* SignalType::Midi;  no-op process() */ };
class VideoChain  : public ProcessingChain { /* SignalType::Video; no-op process() */ };
class FileChain   : public ProcessingChain { /* SignalType::File;  no-op process() */ };

} // namespace sirius
```

### `engine/include/ida/TapeWriter.h` + `.cpp` (NEW, Session 2)

```cpp
namespace sirius {

inline constexpr std::size_t kMaxTapeWriteMessageBytes = 32 * 1024;

struct TapeWriteMessage {
    ChannelId id { 0 };
    Rational lmcTime { 0 };
    std::size_t sampleCount { 0 };
    std::array<std::byte, kMaxTapeWriteMessageBytes> samples {};
};

class TapeWriter {
public:
    TapeWriter (TapeStore& store, const CapabilityTier& tier);
    ~TapeWriter();  // signals shouldExit_, notifies, joins
    // Audio-thread entry; returns false on queue-full (caller reports overload).
    [[nodiscard]] bool tryEnqueue (const TapeWriteMessage&) noexcept;
    // Drain a finalize request through the writer thread before sha256 + rename.
    void flushChannel (ChannelId);

private:
    LockFreeSpscQueue<TapeWriteMessage> queue_;
    std::atomic<bool> shouldExit_ { false };
    // ...condition variable, thread, per-channel file handles, tier flush interval
};

} // namespace sirius
```

## Data flow

```
Audio thread                       Writer thread
─────────────                      ──────────────
audioDeviceIOCallback
  └─ inputMixer.processBuffer
       ├─ for each channel:
       │    └─ channel.processing->process(...)
       └─ for tape-bearing channels:
            └─ tapeWriter.tryEnqueue(message)  ──▶  SPSC queue
                                                       │
                                                       ▼
                                                 writer-thread loop
                                                   ├─ dequeue
                                                   ├─ route by ChannelId
                                                   ├─ append to *.tape.partial
                                                   └─ flush at tier interval

Operator (M22)                     M3 surface
──────────────                     ──────────
arm/disarm gesture ────────────▶   inputMixer.finalizeChannel(channelId)
                                     ├─ tapeWriter.flushChannel(channelId)
                                     ├─ tapeStore.finalize(tapeId)
                                     │    ├─ sha256(partial)
                                     │    └─ rename partial → <sha256>.tape
                                     └─ return new content-addressed TapeId
```
