# M7 S6 — CARemoteLayer cross-process GPU compositing via XPC Mach-port handoff — Design

**Date:** 2026-05-18
**Scope:** Replace the S5 placeholder editor surface with real cross-process GPU compositing. Engine's `CARemoteLayerServer.sharedServer.serverPort` is brokered to each `ida_plugin_host` child through an XPC service bundled in `IDA.app/Contents/XPCServices/`. The child constructs a `CARemoteLayerClient`, hands its plug-in editor's root `CALayer` to the client, and publishes the client's `clientId` back to the engine via the existing `PluginGuiState` shm. The engine wraps `+[CALayer layerWithRemoteClientId:]` inside the existing `OutOfProcessEditorView` NSView. No engine-facing API changes; only the meaning of `responseContextId` flips from placeholder counter to real `CARemoteLayerClient.clientId`.
**Status:** Brainstorm complete; awaiting implementation plan.
**Plan reference:** `docs/superpowers/plans/2026-05-17-v7-alignment.md` lines 487–554 (M7), sandbox/entitlement note at line 550.
**Prior session:** Closes the "CARemoteLayer Mach-port handoff DEFERRED" line item recorded in `continue.md` (S5 commit `52fcd7f`).

## Problem

S5 shipped the complete editor lifecycle and IPC contracts — `PluginGuiState` atomic-seq shared region, `clap_gui_cocoa` create/show/set_parent/hide/destroy bracketing in the child, supervisor-restart re-publication, and a `juce::NSViewComponent`-backed `OutOfProcessEditorView` polling the slot's CAContextID at 30 Hz. What S5 deliberately did *not* land is the actual cross-process pixel transport: `gui_cocoa.mm` publishes a process-unique counter as the contextId, and `OutOfProcessEditorView.mm` renders a tinted placeholder NSView whose colour shifts with the contextId. The operator can see "editor surface is wired" eyes-on; the operator cannot yet see the plug-in's own UI.

S6's job is exactly the pixel transport. Apple's public path for cross-process layer compositing is `CARemoteLayerServer` (engine, singleton) + `CARemoteLayerClient` (child, per-instance). Construction requires the child to receive the engine's `serverPort` as a Mach send-right. The legacy `NSMachBootstrapServer` path was removed in macOS 10.10; the modern path is XPC, which has enough setup overhead (bundled XPC service, launchd registration, connection lifecycle) to deserve its own session.

The acceptance bar is small and concrete: the operator launches `IDA.app`, the test harness wires the synthetic plug-in into the editor view, and the operator sees the synthetic plug-in's coloured Cocoa NSView rendered inside the engine's window via the window-server's compositing — no pixel copying on either side, no engine-thread per-frame work.

## Design principle: zero new engine-facing API surface

S5's `OutOfProcessEffectChainHost::requestEditorShow / Hide / Resize / editorCaContextId / editorSize` are the public engine surface. S6 changes the *value* the slot publishes (placeholder counter → real `CARemoteLayerClient.clientId`) without changing the shape. `OutOfProcessEditorView`'s 30 Hz polling already handles the supervisor-restart re-publication path — that mechanism transparently covers the new "real contextId arrives a few ms after spawn" cadence too.

The XPC plumbing is internal: a singleton `PluginGuiBridge` inside `host/` owns the engine-side XPC connection; the child binary opens its own XPC connection inside `host_process/main.cpp` before publishing the GUI region. Both sides treat XPC as a one-shot port handoff at startup — no XPC traffic on the editor hot path. After bootstrap, CARemoteLayer's window-server compositing carries every pixel; the XPC connections sit idle until child shutdown.

## Operator decisions locked in this brainstorm (2026-05-18)

Six decisions were locked. The implementation plan reads them as source-of-truth.

1. **XPC service bundled inside the .app** (continue.md S6 lean (a)). Path: `IDA.app/Contents/XPCServices/com.larryseyer.ida.gui-bridge.xpc/`. The alternative — anonymous Mach-port pair over a UNIX `socketpair` with port descriptors — needs more bespoke `mach_msg` framing than an XPC service buys per line of code, and the XPC path stays valid once the eventual sandbox tightening lands. The bridge binary is small (~150 LOC) and JUCE-free; CMake builds it as a sibling of `ida_plugin_host`.
2. **Shared XPC service, per-instance XPC connection** (continue.md S6 lean). Bundled XPC services inside the .app are looked up by their .xpc bundle's `CFBundleIdentifier`; no separate `MachServices` Info.plist key is required (that key is for system LaunchAgents/Daemons). The engine holds one shared `xpc_connection_t` to the bridge; each child opens its own `xpc_connection_t` at startup. Per-instance connection gives the supervisor a clean teardown hook on restart (cancel the connection when the child reaps), and matches the per-instance pattern already used for shm regions.
3. **Bridge is a Mach-port broker only** — it caches the most-recently-registered `mach_port_t` (engine's `CARemoteLayerServer.sharedServer.serverPort`) and hands a send-right to each child that asks. No layer state, no per-instance state, no protocol beyond `set_server_port` / `get_server_port`. Mach send-rights are reference-counted by the kernel; the bridge holds one extra send-right while caching, releases it on engine disconnect.
4. **Engine registers the serverPort lazily** at first `configureBus` that adds a slot, not at engine startup. Most sessions never load a plug-in; pre-allocating the `CARemoteLayerServer` singleton (which is a process-wide AppKit/QuartzCore resource) before there's a use for it would burn launch time. The bridge connection itself is also lazy — opens on first registration request.
5. **Child treats missing bridge as soft failure** — if the XPC connection fails (bundle not properly installed, sandbox blocking lookup, bridge crashed and re-spawn pending), the child falls back to the S5 placeholder-counter path and the engine continues to render the tinted placeholder NSView. The supervisor restart logic still works; the operator sees a degraded-but-functional editor surface. No `NotificationLevel::Error` posts for missing GPU compositing — `Warning / PluginEvent` with the failure reason is the chosen surface.
6. **No sandbox entitlements added in S6.** The signed CI workflow is still operator-pending (carryover from M6), and IDA is not yet sandboxed. The bridge's XPC service Info.plist declares `XPCService.ServiceType = Application` (the relaxed type, no sandbox by default), and the bundle's Mach service name uses the app's own bundle ID prefix so future sandbox tightening can add an entitlement scoped narrowly. Document the entitlement diff in `docs/operator/macos-sandbox.md` for the eventual signing session.

## Why CARemoteLayer over alternatives

Three alternatives surfaced during brainstorm:

- **NSView Mach-port reparenting via CGSConnection (private API).** Used by some older host implementations. Private SPI, undocumented, breaks across macOS versions. Rejected.
- **`CALayerHost` forward declaration (Chromium pattern).** `CALayerHost` is also SPI. The S5 draft explored this and abandoned it for exactly this reason (see continue.md S5 deviation #3).
- **Pixel copying via `CGWindowListCreateImage` or a shared `IOSurface`.** Works without Mach-port handoff, but the engine pays CPU per frame, the child pays GPU→CPU readback per frame, and the latency is at least one frame at refresh rate. Defeats the whole point of out-of-process plug-in hosting (which is to keep GUI work off the engine).

`CARemoteLayer*` is Apple's public, documented, macOS 10.7+ API for exactly this case. The window-server does the compositing on the GPU; engine and child do zero per-frame work. The only cost is one Mach-port handoff at child spawn.

## Architecture

```
IDA.app/
└── Contents/
    ├── Info.plist                       (unchanged)
    ├── MacOS/
    │   ├── IDA                 (engine — opens XPC connection to bridge)
    │   └── ida_plugin_host           (child — opens XPC connection to bridge)
    └── XPCServices/
        └── com.larryseyer.ida.gui-bridge.xpc/
            └── Contents/
                ├── Info.plist           (NEW — declares XPC service)
                └── MacOS/
                    └── sirius_gui_bridge (NEW — Mach-port broker)
```

### Component breakdown

**`xpc_service/sirius_gui_bridge`** — new XPC service binary.

- Single C++ translation unit, ~150 LOC, JUCE-free, AppKit-free (only uses `<xpc/xpc.h>` + `<mach/mach.h>`).
- Entry point: `xpc_main(on_new_connection_handler)`.
- Holds one `mach_port_t cachedServerPort` (initially `MACH_PORT_NULL`) plus a `dispatch_queue_t` serializing access.
- Handles two message ops:
  - `op = "set_server_port"`: payload contains `XPC_TYPE_MACH_SEND` named `"port"`. Replaces `cachedServerPort` (releasing the old one via `mach_port_deallocate` if non-null). Replies with `{"ok": true}`.
  - `op = "get_server_port"`: replies with `{"port": <mach_send_right>}` if cached, `{"error": "not_registered"}` otherwise.
- On connection invalidation (engine or child crash), no state changes — the kernel reclaims any send-rights held in transit through that connection automatically.

**`host/include/ida/PluginGuiBridge.h`** + `.cpp` (NEW) — engine-side connection holder.

- Singleton accessed via `PluginGuiBridge::instance()`. Construction is lazy from `OutOfProcessEffectChainHost::configureBus`.
- Owns one `xpc_connection_t` to `com.larryseyer.ida.gui-bridge`. Opens with `xpc_connection_create_mach_service` + `xpc_connection_resume`. Sets an event handler that logs invalidations to `NotificationBus` at `Warning / PluginEvent`.
- On construction: lazily fetches `[CARemoteLayerServer sharedServer].serverPort` (single Cocoa call, requires the AppKit runtime to be running — guaranteed on the engine since JUCE is already up), wraps it into an XPC dictionary with `XPC_TYPE_MACH_SEND`, sends `op = "set_server_port"`. Re-sends on connection re-establishment.
- No public methods beyond `instance()` and `bool isReady() const noexcept`. The Cocoa side is hidden in a `.mm` companion (`PluginGuiBridge.mm`) using the same `extern "C" sirius_*` shim pattern as `OutOfProcessEditorView.mm` to keep AppKit out of the plain C++ TU.

**`host_process/main.cpp`** (MODIFIED) — child-side bootstrap.

- Before the existing shm attach + CLAP pump setup, opens an XPC connection to `com.larryseyer.ida.gui-bridge`, sends `op = "get_server_port"`, blocks up to 250 ms for the reply.
- On success: extracts the Mach send-right via `xpc_dictionary_get_mach_send`, stores it in a global `mach_port_t g_engineServerPort` accessible from `gui_cocoa.mm` via an `extern "C" mach_port_t sirius_engine_server_port()` accessor.
- On failure (bridge unreachable, no `port` in reply, timeout): logs to stderr (which the engine's child-stderr capture surfaces via `NotificationBus`), leaves `g_engineServerPort = MACH_PORT_NULL`. `ida_gui_show` then takes the S5 placeholder path.
- XPC connection is held for the child's lifetime so the kernel doesn't reclaim the send-right.

**`host_process/gui_cocoa.mm`** (MODIFIED) — child-side `CARemoteLayerClient`.

- `ida_gui_show` flow gains: if `sirius_engine_server_port()` returns a valid port, construct `CARemoteLayerClient* client = [[CARemoteLayerClient alloc] initWithServerPort: port]`, then `client.layer = placeholder.layer` (after the existing `set_parent` + `show` calls — the plug-in has by now installed its CALayer hierarchy into the placeholder).
- The published `contextId` becomes `(std::uint32_t) client.clientId` (the CARemoteLayer-assigned identifier).
- `g_editor` gains a `CARemoteLayerClient* remoteClient` field; `teardownEditor` calls `[remoteClient invalidate]` then `[remoteClient release]` before the existing NSView teardown.
- Resize: `placeholder.frame` change already triggers CALayer geometry update; CARemoteLayer transparently picks up the layer-tree change. No extra IPC.

**`host/src/OutOfProcessEditorView.mm`** (MODIFIED) — engine-side `+layerWithRemoteClientId:`.

- `sirius_make_layerhost_nsview` flow: if `contextId != 0`, replace the placeholder tinted layer with `CALayer* remoteLayer = [CALayer layerWithRemoteClientId: contextId]`, install via `view.layer = remoteLayer`. The view's `wantsLayer = YES` is already set.
- `sirius_update_layerhost_contextid` (called on supervisor restart re-publish): rebuilds the layer from the new clientId. Existing 30 Hz polling in `OutOfProcessEditorView.cpp` already detects the change and calls this shim.
- Fallback: if `contextId == 0` (XPC bridge missing, child fell back), keep the S5 tinted placeholder — operator sees degraded surface without crashes.

**`xpc_service/CMakeLists.txt`** (NEW) — build + install the XPC service bundle.

- macOS-only `if(APPLE)` guard.
- Compiles `main.cpp` to `sirius_gui_bridge` with the XPC framework linked.
- Generates `Info.plist` from a template (declares `XPCService.ServiceType = Application`, `CFBundleIdentifier = com.larryseyer.ida.gui-bridge`, `CFBundlePackageType = XPC!`, `CFBundleExecutable = sirius_gui_bridge`). The CFBundleIdentifier IS the name engine + child pass to `xpc_connection_create_mach_service` — no separate `MachServices` key.
- Custom command to assemble the `.xpc` bundle directory structure at build time.

**`app/CMakeLists.txt`** (MODIFIED) — copy the XPC service into the app bundle.

- After `juce_add_gui_app`, add a custom command tied to `IDA`'s POST_BUILD step that copies the `.xpc` bundle into `$<TARGET_BUNDLE_DIR:IDA>/Contents/XPCServices/`. The same step also copies `ida_plugin_host` into `Contents/MacOS/` (this copy may or may not already happen — the spec confirms it inline during implementation; if absent, S6 adds it because both binaries need to ship with the bundle for the XPC `mach_service` lookup to resolve).
- Xcode-generator code-signing block: extend to sign the XPC service bundle with the same Developer ID identity. Hardened runtime + the existing entitlements file applies to both binaries.

### Data flow at editor first-show

```
Engine (juce message thread)
    OutOfProcessEffectChainHost::configureBus
        PluginGuiBridge::instance()         // lazy first call
            xpc_connection_create_mach_service("com.larryseyer.ida.gui-bridge")
            CARemoteLayerServer.sharedServer.serverPort  → port
            xpc_dictionary_set_mach_send("port", port)
            xpc_connection_send_message(set_server_port)

Bridge (sirius_gui_bridge)
    on_new_connection (engine)
        on_message (set_server_port)
            cachedServerPort = received_port           // mach_port_extract_member
            reply {"ok": true}

Engine spawns child via posix_spawn
    args: --shm-engine=...  --shm-host=...  --shm-gui=...  --clap=<path>

Child (ida_plugin_host main.cpp)
    xpc_connection_create_mach_service("com.larryseyer.ida.gui-bridge")
    xpc_dictionary { "op": "get_server_port" }
    xpc_connection_send_message_with_reply_sync (250 ms timeout)
        Bridge replies {"port": cachedServerPort}
    extract → g_engineServerPort
    (then enter normal CLAP pump loop)

Engine (juce message thread, eventually)
    OutOfProcessEditorView::parentHierarchyChanged (first on-screen)
        requestEditorShow → publish Kind/Width/Height in PluginGuiState shm

Child (gui_cocoa.mm, in OnIdle callback)
    ida_gui_show:
        clap_plugin_gui->create / set_parent (placeholder) / show
        client = [[CARemoteLayerClient alloc] initWithServerPort: g_engineServerPort]
        client.layer = placeholder.layer
        responseContextId = client.clientId          // real CARemoteLayer ID
        responseSeq++

Engine (OutOfProcessEditorView::timerCallback, 30 Hz polling)
    sees responseContextId changed (placeholder → real or 0 → real)
    sirius_update_layerhost_contextid:
        view.layer = [CALayer layerWithRemoteClientId: contextId]
    JUCE composites the wrapper view; window-server takes over compositing
```

### Error model

| Failure | Engine effect | Child effect | Operator-visible |
|---|---|---|---|
| Bridge bundle missing / wrong path | XPC connection event-handler invalidates | Same | `Warning / PluginEvent`: "GUI bridge unavailable — degraded placeholder" |
| Bridge crashes mid-session | Engine connection invalidates; engine reconnects on next show | Existing children keep their cached port (kernel retains the right); new children fall back to S5 path until bridge restarts | `Warning / PluginEvent` once per bridge restart |
| `CARemoteLayerClient` initWithServerPort returns nil (stale port) | Slot publishes contextId = 0 | Re-attempt on next supervisor restart cycle | `Warning / PluginEvent`: "remote layer client construction failed" |
| `+layerWithRemoteClientId:` returns nil (invalid clientId) | Falls back to tinted placeholder layer | Unaffected | `Warning / PluginEvent`: "remote layer rendering failed; using placeholder" |
| Engine `CARemoteLayerServer.sharedServer` not available (e.g., headless tests) | `PluginGuiBridge::isReady()` returns false; no XPC traffic | Children proceed with S5 placeholder path | None (expected in headless tests) |

All five failure modes are bounded — none kill the engine, none kill the child, none break the audio thread, none flip `permanentlyBypassed_`. The supervisor's existing 16-miss / 3-restart escalation ladder for audio-thread failures is orthogonal and stays unchanged.

### Test strategy

The XPC `MachServices` lookup only succeeds when the engine binary runs from inside an .app bundle whose `Contents/XPCServices/` includes the bridge. Bare-binary tests run from `build/` have no bundle, so launchd refuses the lookup and `PluginGuiBridge::isReady()` returns false. This is exactly what the soft-failure design accommodates.

The test matrix accordingly splits across three tiers:

1. **Headless unit tests (CI)** — `[plugin-editor-xpc][unit]` tag. Mock `PluginGuiBridge` via a build-time injection point: `PluginGuiBridge::setInstanceForTesting(IGuiBridge*)`. In-process `IGuiBridge` stub records calls. Verify engine-side wiring (lazy first-construct, re-send on reconnect, isReady() reflects connection state).
2. **In-process CARemoteLayer round-trip (CI on Apple Silicon)** — `[plugin-editor-xpc][in-process]` tag. The test constructs `CARemoteLayerServer.sharedServer` and `CARemoteLayerClient` in the SAME process (Apple's API allows this), assigns a known root `CALayer` to the client, verifies `+[CALayer layerWithRemoteClientId:]` returns a non-nil layer whose tree contains a reference to the client. Skipped (`SKIP`, not failed) on non-Apple platforms.
3. **End-to-end operator-launch eyes-on** — manual, gated on the operator running the .app from Finder or `open`. The operator wires the synthetic plug-in into the editor view, observes a coloured Cocoa NSView (synthetic plug-in's content) rendered inside the engine's window. Documented as a step in `docs/operator/m7-eyes-on.md` (new). Eyes-on is the only way to prove the XPC + CARemoteLayer + window-server compositing path works end-to-end; CI can prove every piece *except* that final integration.

Target test count after S6: **~381** (378 from S5 + 3 new `[plugin-editor-xpc]` cases: bridge-lazy-init, in-process-roundtrip, fallback-on-missing-bridge).

`RT_SAFETY_CONTRACT.md` §6 is unchanged. The XPC traffic is engine message thread + child main thread only; no audio-thread surface added. The `[plugin-ipc][.rt-smoke]` regression check must still hold at the S5 71µs / 139µs p99 baseline.

## Risks + mitigations

- **CARemoteLayer's macOS version coverage** — public since 10.7; IDA's deployment target is macOS 13+. No risk.
- **Mach-port send-right lifetime mismatches** — the kernel reference-counts send-rights, but it's easy to leak (every `extract` makes a new right that must be `deallocate`d). Mitigation: every send-right traversal in the spec lists the corresponding `mach_port_deallocate` call site. Code review (mandatory before commit per S4 pattern) re-verifies each pairing.
- **XPC service auto-launch latency** — launchd brings the XPC service up on first connect; first-show latency includes that cold-start. Mitigation: bridge is trivial (~150 LOC, no heavy init); measured cold-start typically < 50 ms on Apple Silicon. The 250 ms child timeout has plenty of headroom.
- **Notarization / Gatekeeper of the XPC bundle** — XPC services inside an app bundle must be signed by the same Team ID as the parent app. Mitigation: extend the existing Developer ID signing block in `app/CMakeLists.txt`. SIP / Gatekeeper acceptance is verified by the notarization step that already exists in the Xcode-generator path.
- **`+[CALayer layerWithRemoteClientId:]` reuse across clientIds** — Apple docs are vague on whether a stale clientId returns a layer that draws blank or returns nil. Mitigation: defensive — if the polled contextId is 0 or unchanged, skip the layer rebuild; only rebuild on new non-zero contextId. The existing S5 polling already implements this.

## One-paragraph summary for the implementation plan

S6 lights up cross-process GPU compositing by adding a bundled XPC service that brokers a single Mach send-right (the engine's `CARemoteLayerServer.sharedServer.serverPort`) from engine to each `ida_plugin_host` child. The child constructs a `CARemoteLayerClient` from that port and publishes its `clientId` as the contextId already plumbed through S5's `PluginGuiState` shm. The engine's `OutOfProcessEditorView` swaps the S5 placeholder layer for `+[CALayer layerWithRemoteClientId:]` whose render is composited by the window-server with zero per-frame engine work. No engine-facing API changes; soft-fails to the S5 placeholder when the bridge is missing. Three new tests cover bridge lifecycle, in-process round-trip, and fallback; operator eyes-on validates the end-to-end pixel path. Target test count: ~381 / ~381.
