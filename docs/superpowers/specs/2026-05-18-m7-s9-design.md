# M7 S9 — Reaper-style separate plug-in windows — Design

**Date:** 2026-05-18
**Scope:** Stop trying to embed the child's plug-in pixels INSIDE the engine's main window. ## Problem

The S5-S8 architecture committed to embedding the child's plug-in pixels inside the engine's main window via Apple's `CARemoteLayerServer` / `CARemoteLayerClient` API. That requires handing the engine's `serverPort` (a mach send-right) from the engine process to each child it spawns. Every public no-entitlement path for that handoff was empirically refuted this session:

| Path | Evidence of refutation |
|------|------------------------|
| In-app `Contents/XPCServices/` bridge (S6-S8) | Per-process-private; launchd spawns a separate bridge per requester. M7 S8 confirmed engine→bridge works but child→bridge cannot see the same instance. |
| `MachServices` in app Info.plist | `MachServices` is a `launchd.plist(5)` key, not an app Info.plist key. App Info.plist is parsed by LaunchServices, not launchd. |
| socketpair + SCM_RIGHTS for mach port | `sendmsg` returns `errno=EBADF` (PoC at `/tmp/scm_test.c`). XNU's SCM_RIGHTS validates each cmsg word as an fd-table entry; mach port names that aren't also valid fds are rejected. |
| Anonymous `xpc_endpoint_t` over inherited fd | No public flat-byte serialization API. The endpoint only survives transport inside an existing xpc_dictionary_t. |
| NSXPCListenerEndpoint via NSKeyedArchiver | Runtime exception (PoC at `/tmp/xpc_poc/`): `*** -[NSXPCListenerEndpoint encodeWithCoder:]: This class may only be encoded by an NSXPCCoder.` |

The remaining viable path within the original architecture was SMAppService-bundled LaunchAgent — keep the bridge process, ship its .plist inside the .app, auto-register on first launch. Functional but reintroduces a broker process and a launchd registration step.

A fundamentally simpler alternative surfaced during operator review: **the embedded-compositing requirement itself is optional.** Every other professional DAW on macOS ships plug-ins as separate floating windows owned by the host or sidecar process; users expect this UX. If the child owns its own NSWindow, the entire mach-port-transfer problem disappears.

## Architecture

```
Engine (Sirius Looper.app main process)
  MainComponent::PluginsPane "Open editor" click
    → MainComponent::openPluginEditor(busId, descriptor)
        → OutOfProcessEffectChainHost::requestEditorShow(slot, w, h)
            → writes PluginGuiState shm: requestKind=Show, requestSeq++
              (existing M7 S5 IPC contract, unchanged)
  Engine reads back via PluginGuiState shm:
    - responseSeq      (poll cadence: existing 30 Hz timer)
    - responseContextId (now: 0 = closed, 1 = open; was: CALayer id)
    - responseWidth / responseHeight (current window size)
  No engine-side NSWindow / NSView for the plug-in. The PluginsPane
  button text toggles "Open editor" / "Close editor" based on
  responseContextId.

Child (sirius_plugin_host)
  main() startup (Apple-only):
    @autoreleasepool {
      [NSApplication sharedApplication];
      NSApp.activationPolicy = NSApplicationActivationPolicyAccessory;
      [NSApp finishLaunching];
    }
  Pump loop (per audio buffer iteration):
    1. Process audio shm ring (unchanged)
    2. Drain pending AppKit events (non-blocking, ~µs cost):
         while (NSEvent* e = [NSApp nextEventMatchingMask:NSEventMaskAny
                                                untilDate:[NSDate distantPast]
                                                   inMode:NSDefaultRunLoopMode
                                                  dequeue:YES])
           [NSApp sendEvent:e];
    3. Service GUI requests (existing path)
  sirius_gui_show(plugin, w, h):
    - Create NSWindow with NSTitledWindowMask | NSClosableWindowMask
      | NSResizableWindowMask, size w×h, title = plug-in name
    - Set NSWindowDelegate so windowWillClose: signals editor-closed
      back to engine via PluginGuiState shm
    - Call plug-in's clap_gui_cocoa.create(window.contentView)
    - Call plug-in's clap_gui_cocoa.show()
    - [window makeKeyAndOrderFront:nil]
    - Publish responseContextId=1 (non-zero = "editor is open")
  sirius_gui_hide(plugin):
    - Call plug-in's clap_gui_cocoa.hide() + destroy()
    - [window close]
    - Publish responseContextId=0
  Window close button (user-driven):
    - NSWindowDelegate windowWillClose:
    - Tear down plug-in editor
    - Publish responseContextId=0
    - (Note: engine polls responseContextId; sees the 0, updates
      its PluginsPane button text from "Close editor" back to
      "Open editor")
```

**Key insight: the M7 S5 PluginGuiState shm IPC contract is unchanged.** The wire still carries requestKind, requestSeq, responseContextId, responseSeq. Only the SEMANTICS of responseContextId shift: it was "CARemoteLayer client id"; now it's just a boolean-ish "editor is open right now" (non-zero) or "editor closed" (0). The engine's 30 Hz polling cadence + supervisor restart logic both continue working untouched.

## Why this is the right architecture

1. **It actually works on modern macOS.** Every other path was refuted empirically this session.
2. **It matches industry-standard DAW UX.** Logic, Live, Reaper, Studio One, FL Studio all ship out-of-process or separate-window plug-ins as independent NSWindows. Users expect this — independently movable, independently sized, can go to different monitors / Spaces.
3. **It's a net code DELETION.** ~700 LOC removed across xpc_service/, PluginGuiBridge*, OutOfProcessEditorView*, the related CMake plumbing, the related tests.
4. **It preserves every other M7 win** — crash isolation, supervisor restart, audio routing, parameter automation, the editor lifecycle IPC, the M7 S7 "Open editor" PluginsPane button.
5. **It's reversible.** A future session could add an opt-in "embed plug-in windows in main window" preference using SMAppService once the mach port handoff becomes worth the operational complexity.

## What this gives up

The plug-in's UI window cannot visually live INSIDE the Sirius Looper main window. It's a sibling top-level window. The S7 work that landed `PluginEditorWindow` (`juce::DocumentWindow` per plug-in) is deleted — that window's only purpose was to host an `OutOfProcessEditorView`, which is also deleted.

## Threading + AppKit event handling in the child

The child process is currently single-threaded — main() runs the pump loop on the only thread. For an NSWindow to receive mouse/keyboard events, the AppKit run loop has to dispatch them. We don't need a full `[NSApp run]` (which would block); we just need to **drain pending AppKit events between audio iterations**:

```objc
@autoreleasepool {
  NSEvent* e;
  while ((e = [NSApp nextEventMatchingMask:NSEventMaskAny
                                untilDate:[NSDate distantPast]
                                   inMode:NSDefaultRunLoopMode
                                  dequeue:YES]) != nil) {
    [NSApp sendEvent:e];
  }
}
```

`[NSDate distantPast]` makes the call non-blocking — returns immediately if no events queued. Empirically the cost is sub-microsecond when the event queue is empty. Compared to a 1024-frame audio buffer at 48 kHz (~21 ms), the overhead is invisible.

`NSApplicationActivationPolicyAccessory` keeps the child out of the Dock and Cmd-Tab while still allowing its NSWindows to receive focus and events. Reaper uses this pattern for its sidecar plug-in hosts.

CLAP's `clap_plugin_gui` extension is documented main-thread-only; that already aligns with where AppKit work needs to happen.

## Soft-failure contract preserved

- If the plug-in's CLAP `gui` extension is missing or doesn't support `CLAP_WINDOW_API_COCOA`, `sirius_gui_show` returns 0 (responseContextId=0); the engine sees "editor failed to open", the PluginsPane button stays as "Open editor" (no toggle). This matches the existing M7 S7 failed-to-load overlay timing (2 s deadline), except now the failure surface lives on the PluginsPane button rather than inside a placeholder NSView.
- If the child crashes, the supervisor restart logic from M7 S4 re-spawns sirius_plugin_host. The engine's polling cycle re-sees responseContextId=0 (fresh process, no editor), button reverts.

## Risk register

- **NSWindow event delivery without full `[NSApp run]`.** The non-blocking drain pattern is well-trodden (used by every plug-in host that embeds AppKit inside a non-AppKit pump). Mitigation: eyes-on testing exercises mouse + keyboard against the synthetic CLAP plug-in. If events don't deliver, the fallback is to move the audio pump to a background thread and call `[NSApp run]` on main.
- **NSApplicationActivationPolicyAccessory window focus.** Accessory apps can show windows but the activation rules differ from regular apps. If windows don't take key focus, switch to `NSApplicationActivationPolicyRegular` (gives a Dock icon — less ideal UX but functional).
- **Multi-instance plug-ins.** Currently `g_editor` is a single static slot in gui_cocoa.mm. For S9, fine — the child hosts one plug-in instance per process anyway. Multi-instance per child is M7 out-of-scope.
- **Window position on screen.** Initial window position is OS-chosen (cascade). Future enhancement: engine sends position hint via shm. Out of scope for S9.
