# M7 S9 — Engine-as-server via socketpair + SCM_RIGHTS mach-port handoff — Design

**Date:** 2026-05-18
**Scope:** Replace the standalone `sirius_gui_bridge.xpc` broker (M7 S6) with a direct mach send-right handoff from the engine to each `sirius_plugin_host` child over a UNIX socketpair inherited at `posix_spawn` time. Delete the entire `xpc_service/` subtree.

## Problem

M7 S8 unblocked the engine→bridge XPC handshake (Developer-ID re-sign + `xpc_connection_create` API fix + os_log instrumentation), but child→bridge cannot work under the bundled-XPC-service model: launchd spawns a **separate** bridge process per requester with empty cached state. In-app XPC services in `Contents/XPCServices/` are PRIVATE to each calling process's bundle scope — they cannot serve as a shared broker. The S6 design's "one .xpc, both engine and child connect, broker holds the cached port" assumption is fundamentally incompatible with Apple's in-app XPC model.

S9's job is to deliver the engine's `CARemoteLayerServer.sharedServer.serverPort` mach send-right to each child without a launchd-registered broker and without bundling a separate XPC service.

## Architectures considered

| Path | Verdict | Why |
|------|---------|-----|
| A. Named MachService via app `Info.plist` | **Refuted** | `MachServices` is a `launchd.plist(5)` key. App Info.plist is parsed by LaunchServices, not launchd. Connection returns `XPC_ERROR_CONNECTION_INVALID` on first send. Requires the LaunchAgent install step explicitly forbidden by the operator. |
| B. Anonymous `xpc_endpoint_t` over inherited fd | **Refuted** | `xpc_endpoint_t` is an `xpc_object_t`, not flat-serializable. Only travels inside an existing `xpc_dictionary_t`. Chicken-and-egg: to pass the endpoint you need a channel; to get a channel you need the endpoint. |
| C. Socketpair + `SCM_RIGHTS` for the mach send-right | **Chosen** | `unix(4)` on Darwin documents SCM_RIGHTS transfers BOTH file descriptors and mach port rights. The payload (a mach send-right) is exactly what SCM_RIGHTS was designed for; XPC was indirection. |

The architectural shape the operator selected — "engine-as-server, no broker process, no install step" — is preserved by Path C. The "XPC server" framing collapses because the available libxpc APIs cannot satisfy the constraint set; the underlying intent (engine vends the send-right directly to its children, no third-party process) is satisfied by the chosen transport. Future commit messages and code use "mach send-right handoff" instead of "engine-as-XPC-server" because the code contains no XPC.

The S6 design doc (line 29) called out the socketpair-fallback path explicitly: *"the alternative — anonymous Mach-port pair over a UNIX socketpair with port descriptors — needs more bespoke `mach_msg` framing than an XPC service buys per line of code"*. That tradeoff was correct at S6 timing; what S8 discovered is that the XPC path needs MORE framing than the socketpair path once you account for the per-process-bundle isolation. The line-of-code count actually tips the other way.

## Design principle: zero new engine-facing API surface

The `OutOfProcessEffectChainHost::requestEditorShow / editorCaContextId / editorSize` engine surface from S5 is unchanged. The XPC connection internal to S6's `XpcGuiBridge` is replaced by a `LocalGuiBridge` that simply reads `[CARemoteLayerServer sharedServer].serverPort` on demand. The transport that was XPC becomes a one-shot `sendmsg` per child spawn. `OutOfProcessEditorView`'s 30 Hz polling against `PluginGuiBridge::instance().isReady()` keeps working; the readiness now means "engine has its serverPort", which is always true after the first `[CARemoteLayerServer sharedServer]` touch.

Per-child readiness moves to `OutOfProcessPluginInstance::hasDeliveredServerPort()`, flipped after the post-spawn `sendmsg` succeeds. `OutOfProcessEditorView`'s bogus-contextId check (commit `5f384a9`) consults this new accessor through the chain host.

## Architecture

```
Engine (Sirius Looper.app main process)
  OutOfProcessEffectChainHost::configureBus
    └── (void) PluginGuiBridge::instance();              // caches serverPort
  OutOfProcessEffectChainHost::ensureSlot
    └── OutOfProcessPluginInstance::spawn
          1. socketpair(AF_UNIX, SOCK_STREAM, 0, sv);    // sv[0] parent, sv[1] child
          2. posix_spawn_file_actions_adddup2(actions, sv[1], 3);
          3. posix_spawnp(&pid, hostBinary, &actions, ...);
          4. close(sv[1]);                                // parent drops child end
          5. MachPortHandoff::sendServerPort(
                 sv[0],
                 PluginGuiBridge::instance().serverPort());
          6. close(sv[0]);
          7. hasDeliveredServerPort_.store(true);

Child (sirius_plugin_host)
  main() before clap-mode bootstrap:
    └── if (mode == "clap"):
          MachPortHandoff::receiveServerPort(
              fd = 3,
              timeout = 1000ms)
              → optional<mach_port_t>
          stores into g_engineServerPort
          close(3)
    └── gui_cocoa.mm reads g_engineServerPort,
        constructs CARemoteLayerClient,
        publishes real clientId via PluginGuiState shm.
```

**Bridge process deletion.** `xpc_service/` directory removed. `add_subdirectory(xpc_service)` removed from top-level CMakeLists.txt. `app/CMakeLists.txt` drops the `sirius_gui_bridge` dependency + the `Contents/XPCServices/` copy. The Developer-ID re-sign of `sirius_plugin_host` and the final outer-app re-sign (M7 S8 work) **stay**: the embedded helper still needs Developer-ID signing for AMFI.

**No audio-thread risk.** All handoff code runs on the JUCE message thread inside `configureBus` / `spawn`. The audio thread is untouched.

**Multiple concurrent children.** Each `OutOfProcessPluginInstance::spawn` creates its own socketpair + sends its own send-right. No shared parent-side state. Same code path runs per slot (scratch buses 1000+ from S7).

## Soft-failure contract preserved

S6 decision #5 (child treats missing bridge as soft failure) is preserved verbatim:

- `recvmsg` timeout in the child leaves `g_engineServerPort == MACH_PORT_NULL`.
- `gui_cocoa.mm` takes the existing S5 placeholder-counter branch.
- Engine renders the failed-to-load overlay (M7 S7 eyes-on work) once the placeholder counter is detected.
- No crash. No `NotificationLevel::Error`. Warning surface only.

## Entitlements / signing

No new entitlements needed. Socket-based IPC inside a single .app's process tree requires no `com.apple.security.mach-lookup.*` declarations. The mach send-right transferred via SCM_RIGHTS is a kernel-level operation between fork-related processes; AMFI / hardened runtime have no opinion. `get-task-allow=false` for notarization (separate todo entry) is unaffected.

`docs/operator/macos-sandbox.md` drops the entire "mach-lookup entitlement for the bridge" section; replaces with a one-line note that the bootstrap is socketpair-based.

## Risk register

- **SCM_RIGHTS for mach port rights on Darwin** — documented in `unix(4)` but less heavily trodden than fd transfer. Mitigated by the unit test exercising the path in-process before integration.
- **`posix_spawn` vs `fork+execvp`** — current `OutOfProcessPluginInstance.cpp` uses `fork()+execvp()`. Switching to `posix_spawn` to get `posix_spawn_file_actions_t` opens the door to env/cwd/signal-handling subtleties. Mitigation: preserve exact env + argv from the existing path; add only the file-actions block.
- **Child-side fd 3 collision** — fd 3 is the first non-stdio descriptor, conventionally free post-spawn. If a future change adds another inherited fd it must claim a different number and the handoff fd must move in lockstep.

## Why this design over alternatives

The selection ranking at S9 brainstorm time (full analysis in `/Users/larryseyer/.claude/plans/read-continue-and-proceed-clever-starfish.md`):

1. **Operator standing preference:** "default to most professional and elegant." Path C ships the smallest possible bootstrap surface — one `sendmsg`, one `recvmsg`, zero broker processes.
2. **Constraint satisfaction:** Path C is the **only** path that satisfies "no install step + no separate XPC bundle" within libxpc's public API surface.
3. **Code economy:** ~250 LOC net removed across the repo. The S6-S8 XPC plumbing was net additive; S9 reverses that into a simpler shape.
4. **Maintainability:** the transport (UNIX domain socket + SCM_RIGHTS) is older than XPC, more universally understood, and won't be deprecated.

The continue.md / todo.md superseded-by-S9 notes preserve the architectural lineage for future archeology.
