# M7 S7 — MainComponent production wiring of OutOfProcess plug-in editor — Design

**Date:** 2026-05-18
**Scope:** Wire `MainComponent` to actually load + open an editor for a scanned CLAP plug-in via the S3–S6 out-of-process infrastructure (`OutOfProcessEffectChainHost` + `OutOfProcessEditorView`). Operator gets a per-descriptor "Open editor" button in the existing `PluginsPane`; clicking it spawns a `ida_plugin_host` child via `configureBus`, brings up the editor in a floating `juce::DocumentWindow`, and the close-box tears the slot down. Multiple plug-ins can be open simultaneously. **No audio routing through the chain in this session** — chain is GUI-eyes-on only; real per-bus chain integration belongs in its own later session along with the M5 OutputMixer wiring.
**Status:** Brainstorm complete; awaiting implementation plan.
**Plan reference:** Continue.md S7 candidate #1 (MainComponent wiring — strongest follow-on).
**Prior session:** Closes the "MainComponent has no production wiring" carryover line recorded across M5–S6 deviation lists.

## Problem

S3 through S6 built the entire out-of-process plug-in hosting stack: IPC rings (S2/S3), audio-thread pump (S3), supervisor + watchdog (S4), GUI lifecycle + `OutOfProcessEditorView` (S5), real CARemoteLayer cross-process compositing (S6). All of it is reachable only from the integration tests in `tests/`. The launched `IDA.app` has zero operator-accessible path that exercises any of it. The S6 close-out eyes-on procedure (`docs/operator/m7-eyes-on.md`) explicitly notes this gap: there is no "Add plug-in" UI yet.

S7's job is the minimum cut between "tests prove the rails" and "operator sees the train." A button. A window. Done. The architectural shape doesn't need to anticipate the eventual M20+ production "add plug-in to a Constituent's effect chain" workflow — that's a separate session and a separate design problem. Today the only question is whether the IPC + CARemoteLayer plumbing actually paints pixels into a window the operator can see and resize.

## Design principle: scratch chain, GUI-only

The minimal eyes-on cut needs `OutOfProcessEffectChainHost::configureBus(...)` to fire so a `ida_plugin_host` child is spawned and the editor lifecycle runs. `configureBus` takes a `busId`, an `EffectChain`, a host binary path, and a CLAP bundle path. None of those need to mean anything in the audio graph — they just need to be well-formed values that the chain host can act on. So S7 uses **scratch bus IDs** (1000, 1001, 1002…) that nothing in the audio graph references. Audio flows through the existing OutputMixer path unchanged; the loaded plug-in receives no audio and produces no audio. The operator can open the editor, tweak controls, see the GUI compose cross-process — but the engine's audio output is unaffected.

This is intentional. Real per-bus chain integration touches the M5 OutputMixer's `setEffectChainHost` wiring, the InputMixer's tape routing, the audio-thread sample budget per slot, and the operator-facing "add plug-in to bus N" UX. Each of those is a session-sized concern. S7 stays at "the IPC works end-to-end and the operator can see it" and defers the rest.

## Operator decisions locked in this brainstorm (2026-05-18)

Five decisions were locked. The implementation plan reads them as source-of-truth.

1. **Floating `juce::DocumentWindow` per loaded plug-in.** Matches every commercial DAW's plug-in editor convention. Each window is resizable, hosts one `OutOfProcessEditorView`, has a close-box that triggers slot teardown. Window title = plug-in display name.
2. **Multi-load.** Each "Open editor" click constructs a new slot in the chain host. Slot key = monotonically incremented (`busId`, `slotIndex=0`) where `busId` starts at `kScratchBusIdBase = 1000` and bumps per load. Re-loading the same descriptor opens a second instance (not "focus the existing one") — matches DAW convention and exercises the per-instance bridge connection model more aggressively.
3. **"Open editor" button per descriptor row.** The existing `PluginsPane` shows descriptors as read-only text. Replace the `juce::TextEditor` with a `juce::ListBox` whose cells render the descriptor + an "Open editor" button. Single-button per row; no "remove from list" or other affordances (that's M20+ scope).
4. **No audio routing.** The chain host's `pumpSlot` is never called for these slots because the OutputMixer doesn't know about the scratch buses. The plug-in's audio thread inside `ida_plugin_host` runs with empty input buffers (or whatever the host child does when no `tryWriteBytes` arrives — verify during implementation). For the eyes-on goal, this is fine.
5. **`OutOfProcessEffectChainHost` is a `MainComponent` member** (not a global singleton). Lives for the lifetime of the main window. Spawned once in `MainComponent`'s ctor, joined in the dtor. The S6 lazy `PluginGuiBridge::instance()` touch in `configureBus` means the bridge connection opens on the first plug-in load, not at app startup — which is what we want (most sessions never load a plug-in).

## Architecture

```
MainComponent
├── OutOfProcessEffectChainHost effectChainHost_   (new — owns the host)
├── std::vector<std::unique_ptr<PluginEditorWindow>> editorWindows_  (new)
└── PluginsPane (existing, modified)
    └── PluginListBox (replaces TextEditor)
        - One row per scanned descriptor
        - Each row has a [Open editor] button
        - Click → MainComponent::openPluginEditor(descriptor)
            1. Allocate next scratch busId (1000, 1001, ...)
            2. Build an EffectChain with one entry pointing at descriptor's .clap path
            3. effectChainHost_.configureBus(busId, chain, hostBinary, clapBundle)
            4. Construct PluginEditorWindow (juce::DocumentWindow):
                - title = descriptor.name
                - content = new OutOfProcessEditorView(effectChainHost_, busId, /*slot*/ 0)
                - close-button callback = MainComponent::closePluginEditor(busId)
            5. Push window into editorWindows_; setVisible(true)

PluginEditorWindow (new — small juce::DocumentWindow subclass)
├── busId_  (so MainComponent can find + tear down)
├── std::unique_ptr<OutOfProcessEditorView> editorView_
└── closeButtonPressed() → callback into MainComponent

MainComponent::closePluginEditor(busId)
    1. effectChainHost_.configureBus(busId, EffectChain{}, ...) // empty chain → tears slot down
    2. Erase the matching PluginEditorWindow from editorWindows_ (its dtor releases the view)
```

### Component breakdown

**`MainComponent` (`app/MainComponent.{h,cpp}`)** — modified.

- New private members: `ida::OutOfProcessEffectChainHost effectChainHost_`, `std::vector<std::unique_ptr<PluginEditorWindow>> editorWindows_`, `std::int64_t nextScratchBusId_ { kScratchBusIdBase };`.
- New static constant: `static constexpr std::int64_t kScratchBusIdBase = 1000;`.
- New private methods: `openPluginEditor(const PluginDescriptor&)`, `closePluginEditor(std::int64_t busId)`.
- `hostBinary()` helper: returns `juce::File` pointing at `Contents/MacOS/ida_plugin_host` (resolved via `juce::File::getSpecialLocation(juce::File::currentExecutableFile)` + parent dir). Falls back to `nullptr` outside a bundle (dev-loop runs from `build/app/...`), in which case "Open editor" buttons are disabled with a tooltip explaining why.
- ctor wires the bridge: `effectChainHost_.setNotificationSink(...)` so plug-in supervisor events (restart, permanent bypass) post into the existing NotificationBus surface.
- dtor: explicit `editorWindows_.clear()` BEFORE `effectChainHost_` destructs, so all slot teardowns complete while the host is still alive. Then `effectChainHost_` joins its supervisor thread normally.

**`MainComponent::PluginsPane`** — modified.

- Replace `juce::TextEditor descriptorsList_` with `PluginListBox descriptorsListBox_` (new inner class, also in `MainComponent.cpp`).
- `setDescriptors(...)` populates the list box's data model instead of a string.
- `setHostBinary(juce::File)` setter — passes the bundle path through so rows know whether to enable their "Open editor" buttons.
- `onOpenEditor` `std::function<void(const PluginDescriptor&)>` callback — wired by `MainComponent` ctor to `[this](const auto& d){ openPluginEditor(d); }`.

**`MainComponent::PluginsPane::PluginListBox`** — new inner class, ~80 LOC.

- Subclasses `juce::ListBoxModel`. Owns the descriptors vector.
- `paintListBoxItem` renders three lines per row: name + manufacturer (bold), uniqueId (mono), filePath (mono, dimmed). Same content the old TextEditor showed, just per-row.
- Each row has a `juce::TextButton` "Open editor" rendered via `refreshComponentForRow` (JUCE's row-component pattern). Button enabled iff host binary is valid; tooltip explains disabled state.
- Failed-files section rendered below the list as a separate `juce::Label` (since failed entries don't get a button).

**`MainComponent::PluginEditorWindow`** — new inner class, ~40 LOC.

- Subclasses `juce::DocumentWindow`.
- Ctor: takes name, busId, and an `OutOfProcessEditorView*` (non-owning view pointer; ownership lives in this window's content component). Sets `setUsingNativeTitleBar(true)`, `setResizable(true, false)`, attaches the view via `setContentOwned`.
- `closeButtonPressed()` invokes a `std::function<void(std::int64_t)>` callback set by `MainComponent` — calls back into `closePluginEditor(busId_)`.
- Initial bounds: read `OutOfProcessEditorView::currentContextId() == 0` initially (no editor shown yet); use a 600×400 default. The editor view's polling timer detects the child's first publish (typically <100 ms after spawn) and rebuilds the embedded NSView at the plug-in's preferred size.

### Lifecycle

```
App launch
    MainComponent ctor
        effectChainHost_ default-constructs (supervisor thread starts)
        PluginGuiBridge::instance() is NOT touched yet
        PluginsPane created, no descriptors
        hostBinary path resolved (might be null in dev-loop)

User scans plug-in folder → existing flow → pluginsPane_->setDescriptors(...)
    PluginListBox repopulates; rows show "Open editor" buttons

User clicks "Open editor" on row N
    MainComponent::openPluginEditor(descriptor)
        busId = nextScratchBusId_++           (e.g. 1000)
        chain = EffectChain with one entry (filePath = descriptor.filePath, bypassed = false)
        effectChainHost_.configureBus(busId, chain, hostBinary, descriptor.filePath)
            ↳ PluginGuiBridge::instance() lazily constructs (first load only) — XPC connection opens
            ↳ Spawns ida_plugin_host child; bootstraps; loads CLAP
        Construct PluginEditorWindow(descriptor.name, busId, new OutOfProcessEditorView(effectChainHost_, busId, 0))
        editorWindows_.push_back(window); window.setVisible(true)
        window's editor view polling timer detects the child's first contextId publish; embedded NSView rebuilds
            → CARemoteLayer composites the plug-in's actual GUI into the window (if .app context),
            OR tinted placeholder (if dev-loop / no XPC bridge)

User clicks the window's close-box
    PluginEditorWindow::closeButtonPressed → callback → MainComponent::closePluginEditor(busId)
        effectChainHost_.configureBus(busId, EffectChain{}, hostBinary, juce::File{})
            ↳ Empty chain → host's stale-slot eraser removes the slot → supervisor reaps the child
        editorWindows_.erase(window)         ↳ OutOfProcessEditorView dtor → ~NSViewComponent → engine-side layer torn down

App shutdown
    MainComponent dtor
        editorWindows_.clear()                ↳ each window closes; each editor view dtor; each slot teardown queued
        effectChainHost_ dtor                 ↳ supervisor thread joined; any pending children reaped
```

### Error model

| Failure | Effect | Operator-visible |
|---|---|---|
| Host binary not found (dev-loop, no .app bundle wrapping) | "Open editor" buttons disabled with tooltip "Open editor unavailable — launch the .app for plug-in hosting" | Tooltip on hover |
| `configureBus` throws (shm exhaustion, etc.) | Caught; window not created; `NotificationLevel::Error` posted | Notification banner |
| Child crashes immediately after spawn | Supervisor's existing 16-miss + 3-restart logic fires; user sees Info posts (restart attempts) then Error (permanent bypass) | Notification banner; editor window shows placeholder until restart succeeds, then real content |
| XPC bridge missing (e.g., XPCServices/ not in bundle) | Editor window shows tinted placeholder; no error posted (degraded but not broken) | Coloured rectangle instead of real plug-in UI |
| Plug-in has no GUI (`clap_plugin_gui` extension absent) | `gui_cocoa.mm`'s `is_api_supported` returns false; `ida_gui_show` returns 0; engine polls forever, never sees a non-zero contextId | Editor window stays at the initial 600×400 with a "(no editor available)" tinted background — actually, this is worth a comment fix; see Risks below |
| Operator opens the same plug-in twice | Both load successfully (separate scratch busIds, separate ida_plugin_host children); two windows, two editors | Two windows |

All paths are bounded; nothing kills the app or wedges the engine. Audio output is uninterrupted regardless of what happens to a plug-in editor (because no audio routes through these scratch chains).

### Testing strategy

Three new headless test cases in `tests/MainComponentPluginEditorTests.cpp` (new file):

1. **`[main-component-plugin-editor]` MainComponent constructs + destructs cleanly with no editor windows open** — verifies the empty-lifecycle case (ctor → dtor, supervisor thread joins).
2. **`[main-component-plugin-editor]` openPluginEditor with synthetic descriptor spawns a child + window** — uses the synthetic test plug-in's .clap bundle (already used by S5 editor tests); asserts `editorWindows_.size() == 1` and `effectChainHost_.childPidForTestingAtSlot(busId, 0) > 0`.
3. **`[main-component-plugin-editor]` closePluginEditor tears down child + window** — opens, then closes; asserts editorWindows_ is empty AND `childPidForTestingAtSlot` returns the negative sentinel (slot erased).

These use the headless JUCE message loop pattern already established by existing MainComponent-adjacent tests (if any — verify during implementation; if none, the tests construct MainComponent inside a `juce::ScopedJuceInitialiser_GUI` and pump the message loop manually). Test count target: **387/387** (384 from S6 + 3 new).

Operator eyes-on remains the only way to verify CARemoteLayer pixels actually paint inside the window (CI can't render); the procedure in `docs/operator/m7-eyes-on.md` becomes properly executable after S7 ships (the step that currently says "M20+ adds a debug menu" can be deleted — the "Open editor" button IS the debug menu).

## Acceptance criteria

- `MainComponent` owns an `OutOfProcessEffectChainHost effectChainHost_` member with proper ctor/dtor lifecycle.
- `PluginsPane` shows scanned descriptors in a `juce::ListBox` with one "Open editor" button per descriptor row (replaces the read-only `TextEditor`).
- Clicking "Open editor" allocates a fresh scratch busId, calls `configureBus`, and opens a `juce::DocumentWindow` containing an `OutOfProcessEditorView`.
- The window is resizable, has a native title bar, and close-button tears down the slot + reaps the child.
- Multiple "Open editor" clicks produce multiple windows + multiple `ida_plugin_host` children (verified at runtime via `ps`).
- Three new `[main-component-plugin-editor]` ctest cases pass.
- Existing 384 ctest cases still pass.
- `RT_SAFETY_CONTRACT.md` unchanged (verified by diff).
- App launch behaviour unchanged when no plug-in is loaded (no XPC bridge, no child, no editor windows).
- Operator eyes-on per `docs/operator/m7-eyes-on.md` (updated): launch .app, scan a folder with a CLAP plug-in, click "Open editor", see the plug-in's actual GUI rendered cross-process inside the floating window.
- App shutdown is clean: all windows close, all children reap, supervisor thread joins, no orphan processes (verified via `ps` after `osascript -e 'quit app "IDA"'`).

## Out of scope (carry forward, NOT touched in S7)

- **Audio routing through the chain** — scratch chains don't feed the OutputMixer; the engine's audio output is untouched. Real per-bus chain integration (operator picks "add plug-in to bus N", chain becomes part of the audio graph) is a session-sized concern that belongs in M5 OutputMixer wiring / M20+ production UX.
- **Parameter automation** — no parameter-display UI, no parameter writes from the engine. The CLAP plug-in's own editor is the operator's surface.
- **State save/load** — loaded plug-ins do not survive app restart. Session persistence of plug-in state is M9+ scope.
- **Presets** — no preset browser, no preset save/load.
- **Multi-slot chains** — every "Open editor" creates a single-slot chain on its own scratch busId. Multi-slot chains arrive with real per-bus chain integration.
- **Bundle re-sign gap** — operator-pending CI signing handoff still owns this (S6 deviation #2).
- **Audio-ring SPSC violation** — S5 carryover, still pending its own session.

## Risks + mitigations

- **JUCE headless test harness for MainComponent.** Existing MainComponent-adjacent tests may or may not exist. If they don't, the three new tests need to set up a minimal `juce::ScopedJuceInitialiser_GUI` and pump the message loop manually to drive the editor lifecycle. Risk: JUCE's GUI message loop interactions with the supervisor thread + XPC bootstrap timeouts could make these tests slow or flaky. Mitigation: if the headless tests prove fragile, narrow the scope to construct/destruct lifecycle only (skip the actual configureBus + child spawn — that path is already covered by `OutOfProcessEditorTests`). Decide during implementation.
- **`hostBinary()` resolution in dev-loop runs.** When MainComponent runs under ctest (no .app wrapper), `juce::File::currentExecutableFile` returns the path to the test binary, not `ida_plugin_host`. The resolution helper must handle this and return an invalid path so the "Open editor" buttons disable themselves. Tests don't depend on the buttons working — only on the host being well-formed for direct `configureBus` calls.
- **Plug-in with no GUI.** A CLAP plug-in without the `clap_plugin_gui` extension causes the child to return 0 from `ida_gui_show`, which the engine polls forever expecting a contextId. The editor window stays at the initial 600×400 tinted placeholder; the operator has no indication that the plug-in has no GUI. Mitigation: after a 2-second timeout with no published contextId, the `OutOfProcessEditorView` could publish a stronger "(no editor available)" signal — but this is a polish item for a later session. For S7, document the behaviour in the operator eyes-on doc.
- **DocumentWindow close-box ordering.** JUCE's `closeButtonPressed()` runs on the message thread; it must not destroy the window from within its own callback (use-after-free in the JUCE event dispatcher). Mitigation: the callback into `MainComponent::closePluginEditor` schedules the window removal via `juce::MessageManager::callAsync`, so the window destructor runs in the next message-loop iteration.
- **Multiple plug-ins of the same descriptor.** Bridge connection per child is fine (S6 design). Scratch busIds are unique. The only concern is operator confusion if two windows have identical titles. Mitigation: append `" (#N)"` to the second and subsequent window titles for the same descriptor.

## One-paragraph summary for the implementation plan

S7 wires `MainComponent` so the operator can finally see S6's cross-process compositing in action. A new `OutOfProcessEffectChainHost effectChainHost_` member lives on `MainComponent`; the existing `PluginsPane` swaps its read-only `juce::TextEditor` for a `juce::ListBox` with "Open editor" buttons per descriptor. Each click spawns a `ida_plugin_host` child on a fresh scratch busId (1000+) and floats a `juce::DocumentWindow` containing an `OutOfProcessEditorView`; close-box tears the slot down. No audio routes through these scratch chains — engine output is uninterrupted. Three new headless lifecycle tests + an updated operator eyes-on doc complete the deliverable. Target test count: ~387 / ~387.
