# M8 S6 — LMC calibration corruption recovery

## Problem

V7 alignment (line 602) defines the acceptance: on session load, validate the
LMC calibration table's checksum; on failure, trigger fresh loopback
calibration via the existing `AudioDeviceCalibration`, emitting a
`NotificationBus` event during recalibration.

The codebase does not yet have the infrastructure that wording assumes:

- **No checksum, no persistence.** `AudioDeviceCalibration`
  (`engine/include/ida/AudioDeviceCalibration.h`) is a pure value type:
  `rateFactor_`, `offsetSeconds_`. `MainComponent::calibration_` is initialized
  to `identity()` at declaration (`app/MainComponent.h:168`) and reset every
  launch — never written to or read from disk.
- **No loopback engine.** No routine anywhere *measures* a fresh calibration.
  That is a net-new real-time DSP feature (emit a probe, measure round-trip
  drift, derive rate + offset) requiring live audio I/O — not buildable or
  verifiable in a headless TDD loop.

The calibration is a per-device property of one hardware clock (white paper
Part 4.3). It belongs to the machine, not to a session document: a session
moved to other hardware must not carry a stale calibration with it.

This session ships the **persistence + checksum + validate-on-load + notify**
layer behind an honest seam, matching the M8 S1/S3 precedent (M8 S3 shipped a
pure `validate()` + caller-side `postConstituentStateNotifications` with the
production data source — a `TapeStore`-backed resolver — deferred). The real
loopback DSP engine is deferred; on corruption we recover to identity, post a
Warning, and persist a `recalibrationPending` marker the future engine consumes.

## Scope

In:

- `CalibrationStore` — a pure, JUCE-free serialize / parse-and-validate module
  for a calibration document (`AudioDeviceCalibration` + `recalibrationPending`),
  carrying a SHA-256 over a canonical payload. String in / string out: no disk,
  exhaustively unit-testable. Mirrors the engine's stated philosophy
  (`engine/CMakeLists.txt`: public API JUCE-free, juce_core is a private .cpp
  detail).
- `postCalibrationRecoveryNotification` — a caller-side helper that posts a
  Warning on a non-`Ok` load result, parallel to
  `postConstituentStateNotifications`. Kept out of the parse so the parse stays
  sink-free and unit-testable.
- `MainComponent` wiring — resolve a device-scoped sidecar path, load + validate
  at startup before the calibration is handed to the audio callback, recover +
  re-persist + notify on a non-`Ok` result, and re-check on session load.

Out (deferred, recorded in `todo.md`):

- **The real loopback DSP calibration engine.** The probe / measure / derive
  routine. The `recalibrationPending` marker is the seam it will consume. Own
  session; needs live audio I/O and operator hardware.
- **Multi-device keying.** The sidecar holds one calibration; keying by audio
  interface identity is a future extension (device-identity plumbing absent).

## Key constraints

- **Engine public API stays JUCE-free.** `CalibrationStore.h` uses only
  `std::string` / std types, core `Rational`, core `INotificationSink`, and
  `AudioDeviceCalibration`. The hand-rolled canonical text format avoids pulling
  a JSON library into the engine's public surface and keeps the parser pure.
- **`AudioDeviceCalibration` is not modified.** It stays an immutable value type.
  `recalibrationPending` is store metadata, never a field on the calibration
  value — identity preserved, surgical.
- **Checksum reuse.** SHA-256 via core `sha256Hex`
  (`core/include/ida/Sha256.h`) over the canonical payload bytes — the same
  fingerprinting primitive `VersionPinningRecord` uses.
- **Recovery is fail-loud.** A `Missing` or `Corrupt` sidecar resets to identity
  AND posts a Warning AND persists `{identity, pending=true}` so nothing silently
  swallows the corruption and the future engine knows a re-measure is owed.

## Design

### Canonical document format

A two-section text document: a canonical payload, then a checksum line over the
exact payload bytes.

```
rateFactor=<num>/<den>
offset=<num>/<den>
pending=<0|1>
sha256=<64-hex>
```

The payload is the first three lines (through the trailing newline after
`pending=...`); `sha256=` is computed over those payload bytes and is not itself
part of the hashed region. Rationals serialize via `Rational::numerator()` /
`denominator()` (already normalized), parse by splitting on `/`. This is a small
internal device file, not operator-facing; the format is deliberately minimal
and the public API is format-agnostic so tests assert behavior, not bytes.

### `CalibrationStore` — `engine/include/ida/CalibrationStore.{h,cpp}`

```cpp
enum class CalibrationLoadStatus { Ok, Missing, Corrupt };

struct CalibrationDocument
{
    AudioDeviceCalibration calibration { AudioDeviceCalibration::identity() };
    bool recalibrationPending { false };
};

/// Canonical payload + SHA-256 wrapper. Pure.
std::string serializeCalibration (const CalibrationDocument&);

struct CalibrationParseResult
{
    CalibrationLoadStatus status;     // Ok | Missing | Corrupt
    CalibrationDocument   document;   // identity + pending=true unless Ok
};

/// Parse and validate. Empty input -> Missing. Malformed, out-of-range, or
/// checksum-mismatched input -> Corrupt. On any non-Ok status the returned
/// document is {identity, pending=true} so the caller can recover and re-persist
/// directly. Pure — no disk, no sink.
CalibrationParseResult parseAndValidateCalibration (const std::string& fileContents);
```

`serializeCalibration` builds the payload, hashes it with `sha256Hex`, appends
the `sha256=` line. `parseAndValidateCalibration` parses the four lines,
recomputes the hash over the payload bytes, compares; any structural failure
(missing line, bad rational, non-positive rate factor — the same invariant the
`AudioDeviceCalibration` ctor enforces — or hash mismatch) yields `Corrupt`;
genuinely empty content yields `Missing`. Non-`Ok` always returns a recoverable
`{identity, pending=true}` document.

### `postCalibrationRecoveryNotification` — same header

```cpp
void postCalibrationRecoveryNotification (CalibrationLoadStatus status,
                                          INotificationSink&    sink);
```

No-op on `Ok`. On `Missing` / `Corrupt`, posts one
`NotificationLevel::Warning`, `Category::StateRepair` (the M8 S1/S3 load-validation
family) with a message naming the cause and the recovery, e.g.
`"calibration corrupt — reset to identity, recalibration pending"` /
`"calibration missing — using identity, recalibration pending"`, within the
128-byte cap. `StateRepair` over `DeviceEvent` keeps load-time recovery in one
notification family with the version-pinning and Constituent-state checks.

### App-layer wiring — `app/MainComponent.cpp`

File I/O lives in the app layer beside the existing session file I/O (small,
integration-only, the same status as the M8 S1/S3 load hooks):

- **Path:** `juce::File::getSpecialLocation(userApplicationDataDirectory)`
  `.getChildFile("IDA/calibration.json")` (the app-data resolution
  pattern from `host/src/ClapScanner.cpp:23`). The `.json` extension is kept for
  familiarity even though the body is the canonical text format.
- **Startup:** before `audioCallback_->setCalibration(&calibration_)`
  (`MainComponent.cpp:872`), read the sidecar (absent file → empty string →
  `Missing`) → `parseAndValidateCalibration`.
  - `Ok` → adopt `document.calibration` into `calibration_`.
  - `Missing` / `Corrupt` → leave `calibration_` at identity, write a fresh
    `serializeCalibration({identity, pending=true})` to the sidecar, and
    `postCalibrationRecoveryNotification`.
  - **Sequencing:** if `notificationBus_` is not yet constructed at line 872,
    cache the status and flush the notification once the bus is wired (the bus is
    a member; confirm its construction order during implementation and post at
    the earliest point it is live).
- **On session load** (`chooseFileAndLoad`, after `deserializeSession`, beside
  the M8 S1/S3 hooks at `MainComponent.cpp:1598–1612`): re-read + re-validate the
  sidecar and `postCalibrationRecoveryNotification` on a non-`Ok` result. Catches
  corruption that happened after launch and satisfies the spec's literal "on
  session load" wording. Reuses the same pure functions — no new logic.

### Data flow

startup → read sidecar → `parseAndValidateCalibration` → (`Ok`) adopt
calibration | (non-`Ok`) identity + re-persist `{identity, pending=true}` +
Warning → `setCalibration`. Session load → re-read sidecar → re-validate →
Warning on non-`Ok`. The loopback engine that will clear `recalibrationPending`
is M8+ future work.

## Testing — `tests/CalibrationRecoveryTests.cpp`, tag `[calibration-recovery]`

1. **Round-trip.** `serializeCalibration` of a non-identity calibration
   (e.g. `rateFactor 999987/1000000`, `offset 1/48000`, `pending=false`) →
   `parseAndValidateCalibration` → `Ok`, calibration equals the original
   exactly (exact `Rational` equality), `recalibrationPending` preserved.
2. **Pending round-trips.** A `{identity, pending=true}` document round-trips to
   `Ok` with `pending=true`.
3. **Tampered payload → Corrupt.** Flip a digit in the `rateFactor` line without
   updating `sha256=` → `Corrupt`; returned document is `{identity,
   pending=true}`.
4. **Tampered checksum → Corrupt.** Mutate the `sha256=` hex → `Corrupt`.
5. **Malformed → Corrupt.** Missing a line / non-numeric rational / zero
   denominator / non-positive rate factor → `Corrupt`.
6. **Empty → Missing.** Empty string → `Missing`, document `{identity,
   pending=true}`.
7. **Notify on non-Ok.** With a recording `INotificationSink`,
   `postCalibrationRecoveryNotification` posts exactly one
   `Warning`/`StateRepair` for `Corrupt` and for `Missing`, and posts nothing for
   `Ok`. (Mirrors the `postConstituentStateNotifications` test style.)

`MainComponent` startup/load wiring is integration glue (file I/O + ctor
sequencing), verified by clean build + ctest staying green + eyes-on, not unit
tested — the same status as the M8 S1/S3 load hooks.
